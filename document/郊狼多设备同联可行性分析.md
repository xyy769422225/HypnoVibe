> 版本：v1.0

## 1. 结论

**可行，但有明确的工程约束。** 理论上可以同时驱动 3-4 台郊狼设备，实际推荐上限 2-3 台。主要瓶颈不在协议本身，而在 Android BLE 的并发写入能力和 100ms 时序精度。

| 维度 | 评估 |
|------|------|
| 协议层面 | 每个郊狼完全独立，无互斥/互锁机制 |
| BLE 并发 | 支持多连接，但有 GATT 操作串行化限制 |
| 时序精度 | 2 台可保证 100ms，4 台以上可能出现漂移 |
| 推荐上限 | **3 台**（保守），**4 台**（需验证实测） |

## 2. Android BLE 多连接能力

### 2.1 系统限制

| 限制项 | 数值 | 来源 |
|--------|:--:|------|
| Android BLE 同时连接数上限 | **7 个** (Android 5.0+) / **4 个** (部分 OEM) | `BluetoothAdapter` 文档 |
| 单连接 MTU | 默认 23 bytes，可协商至 **144 bytes** | BLE 4.2+ |
| GATT 操作串行化 | **同一 GATT 实例的请求必须串行**，但不同设备的 GATT 可并行 | Android BLE 最佳实践 |
| 单次 writeCharacteristic 耗时 | 通常 **10-30ms**（含 ATT 层确认） | 经验值 |

### 2.2 多设备写入时序

郊狼 V3 的 B0 指令为 20 bytes，在默认 MTU=23 下**单包即可发送**（无需分包），这是一个有利条件。

```
单设备 B0 写入耗时: ~10-20ms (BLE GATT write + ATT ACK)
双设备顺序写入:    ~20-40ms (在 100ms 窗口内，安全)
三设备顺序写入:    ~30-60ms (在 100ms 窗口内，可行)
四设备顺序写入:    ~40-80ms (接近临界，偶有超时风险)
```

> 关键约束：所有设备的 B0 指令必须在**同一个 100ms 窗口**内发出，否则不同设备的输出会有相位差。

### 2.3 并行写入 vs 顺序写入

Android BLE 的 GATT 写入对**不同设备**理论上可以并行，但实践中：

- Android BLE 栈在 `bluetooth` 进程中处理所有 GATT 操作，内部有排队
- 即使多线程分别 writeCharacteristic，底层仍是串行化处理
- 推荐做法：**在一个专用 BLE 线程中顺序写入所有设备**，避免竟态

## 3. 郊狼协议层面的多设备分析

### 3.1 协议独立性

每个郊狼设备是完全独立的：
- 独立的 BLE 连接（MAC 地址不同）
- 独立的 GATT 实例
- 独立的 B0 指令、BF 指令
- 独立的 B1 反馈

不存在协议层面的互斥或依赖关系。

### 3.2 序列号机制在多设备下的影响

每个设备的序列号独立维护。假设同时修改两台设备的强度：

```
设备1: 发送 B0(seq=1, mode=绝对, A=80) → 等待 B1(seq=1)
设备2: 发送 B0(seq=1, mode=绝对, B=60) → 等待 B1(seq=1)
```

两者互不干扰，因为 B1 是每个设备的 Notify 独立上报的。

### 3.3 软上限的影响

BF 指令每个设备独立设置。多设备场景下，不同设备可以有不同的软上限。

## 4. 线程模型与定时器设计

### 4.1 现有架构的线程模型

```
Audio 线程 (C, URGENT_AUDIO):  ~30Hz 回调 onPositionUpdate
BLE 线程 (Java, DEFAULT):      GATT 写入、扫描回调
```

### 4.2 多设备适配方案

```
                    Audio 线程 (~33ms)
                         │
                         ▼ onPositionUpdate(ms)
              ┌──────────────────────┐
              │  TimelineEngine (C)  │
              │  计算各通道强度快照   │
              └──────────┬───────────┘
                         │ Map<ch, strength>  (~33ms)
                         ▼
              ┌──────────────────────┐
              │  强度累积缓冲 (Java) │  ← 累积3帧 ≈ 100ms
              │  CoyoteCoordinator  │
              └──────────┬───────────┘
                         │ 每100ms触发一次
                         ▼
    ┌─────────┬─────────┬─────────┐
    │ 郊狼#1  │ 郊狼#2  │ 郊狼#3  │   顺序写入
    │ B0 cmd  │ B0 cmd  │ B0 cmd  │   (BLE线程)
    └─────────┴─────────┴─────────┘
```

**关键设计**：Audio 线程回调频率（~30Hz ≈ 33ms）与协议周期（100ms）不匹配，需要在 Java 层做 3 帧累积缓冲，然后每 100ms 一次性输出给所有设备。

### 4.3 定时器方案

```java
// 方案A：独立定时器（推荐）
// 每个 CoyoteDevice 使用 ScheduledExecutorService 的 100ms 定时任务
// 优点：隔离性好，一个设备断开不影响其他
// 缺点：多个定时器之间可能产生微小漂移

// 方案B：统一定时器（更精确）
// 一个全局 100ms 定时器回调 → 依次写入所有设备
// 优点：无漂移，时序最精确
// 缺点：耦合度高
class CoyoteCoordinator {
    private final List<CoyoteDevice> devices = new ArrayList<>();
    private final ScheduledExecutorService scheduler = 
        Executors.newSingleThreadScheduledExecutor();
    
    public void start() {
        scheduler.scheduleAtFixedRate(() -> {
            // 1. 从 TimelineEngine 拉取当前 100ms 窗口的波形数据
            WaveformWindow window = engine.nextWindow();
            
            // 2. 依次写入所有设备
            for (CoyoteDevice device : devices) {
                if (device.isConnected()) {
                    device.writeB0Window(window);
                }
            }
        }, 0, 100, TimeUnit.MILLISECONDS);
    }
}
```

> 推荐**方案 B（统一定时器）**，因为 HypnoVibe 的播放场景要求所有设备严格同步。

## 5. 资源消耗估算

### 5.1 内存

| 资源 | 单设备 | 3设备 |
|------|:---:|:---:|
| CoyoteDevice 实例 | ~2KB | ~6KB |
| B0 指令缓冲 (20B) | 20B | 60B |
| GATT 实例 (系统) | ~4KB | ~12KB |
| **合计** | **~6KB** | **~18KB** |

内存开销极小，不是瓶颈。

### 5.2 BLE 带宽

| 项目 | 单设备 | 3设备 |
|------|:---:|:---:|
| B0 写入 (20B × 10次/秒) | 200 B/s | 600 B/s |
| B1 接收 (4B × 10次/秒 max) | 40 B/s | 120 B/s |
| 连接维护开销 | ~50 B/s | ~150 B/s |
| **合计** | **~300 B/s** | **~900 B/s** |

BLE 物理层带宽约 100-300 KB/s（取决于 PHY），协议开销远未触及带宽上限。

### 5.3 CPU

| 项目 | 单设备 | 3设备 |
|------|:---:|:---:|
| B0 指令构造 | <0.1ms | <0.3ms |
| B1 消息解析 | <0.1ms | <0.3ms |
| GATT writeCharacteristic | 10-20ms | 30-60ms (串行) |

CPU 开销主要是 GATT 写入等待（I/O bound），不占用 CPU 计算资源。

## 6. 集成到现有架构的改造点

### 6.1 DeviceProtocolAdapter 接口需要修改

当前设计：

```java
interface DeviceProtocolAdapter {
    fun encodeCommand(channelId: String, strength: Int): ByteArray  // 单通道
}
```

郊狼 B0 是双通道合并指令，需要改为：

```java
// 方案：批量编码接口
interface DeviceProtocolAdapter {
    /** 编码一个完整输出周期的指令 */
    fun encodeCycleCommand(
        channelStrengths: Map<String, Int>,     // 所有通道当前强度
        channelWaveforms: Map<String, WaveSlot[]> // 所有通道波形数据（4×25ms）
    ): ByteArray
    
    fun encodeStopCommand(): ByteArray
    fun getServiceUUID(): UUID
    fun getWriteCharUUID(): UUID
    fun getNotifyCharUUID(): UUID
}
```

其中 `WaveSlot` 同时包含频率和强度：

```java
class WaveSlot {
    int frequency;  // 协议值 10-240
    int strength;   // 0-100
}
```

### 6.2 WaveformFile 模型需要扩展

当前设计只有强度：

```json
"dataPoints": [{ "timeMs": 0, "strength": 10 }]
```

需要支持频率维度：

```json
"dataPoints": [{ "timeMs": 0, "frequency": 100, "strength": 10 }]
```

或者为郊狼格式保留原始数据：

```json
{
  "deviceType": "coyote",
  "sourceFormat": "dg_lab_v3",
  "rawFrames": [
    {"freq": 10, "str": 0},
    {"freq": 10, "str": 20}
  ]
}
```

### 6.3 指令节流需要区分设备类型

当前设计：

```
最小发送间隔: 50ms（全局）
```

郊狼的实际约束：

```
固定发送间隔: 100ms（硬约束）
可配置项: 无（郊狼不支持调整输出窗口）
强度变化阈值: ≥ 1（协议粒度允许）
```

建议将节流参数从全局移到设备类型级别：

```java
class DeviceThrottleConfig {
    long minIntervalMs;      // 郊狼=100, Lovense=可配置
    int strengthThreshold;   // 郊狼=1
    boolean fixedInterval;   // 郊狼=true（不可变）
}
```

### 6.4 TimelineEngine 需要输出频率维度

当前架构：

```
TimelineCalc (C) → 强度快照 Map<ch, Int>
```

需要扩展为：

```
TimelineCalc (C) → 输出快照 Map<ch, WaveSlot[4]>
                     每个槽位 = {频率, 强度}
```

## 7. 风险与边界条件

### 7.1 已知风险

| 风险 | 等级 | 缓解措施 |
|------|:--:|------|
| BLE 断连导致设备间时序不同步 | 中 | 断连设备立即从写入列表移除，不影响其他设备 |
| 100ms 窗口内写入超时 | 低 | 监控写入耗时，超过 80ms 时降级（跳过该周期或减少设备数） |
| 不同 Android 厂商 BLE 栈差异 | 中 | 连接数上限保守取 4；需在主流机型上实测 |
| GATT 写入队列堆积 | 低 | 使用 `WRITE_TYPE_NO_RESPONSE` 可减少 ATT 层等待 |
| 设备固件版本不一致 | 低 | 连接时检查 B1 返回的强度范围，自适应 |

### 7.2 不推荐的做法

- **不要用多个线程分别写不同设备**：Android BLE 栈内部串行化，多线程只会增加竟态
- **不要用 `WRITE_TYPE_NO_RESPONSE` 代替默认写入**：虽然更快，但 B0 指令需要确认送达（丢帧会导致波形不连续）
- **不要把 100ms 定时器放在 UI 线程**：必须用独立线程或 ScheduledExecutor

### 7.3 推荐连接流程（多设备）

```
1. 用户选择设备类型 → 扫描 → 逐个连接
2. 每个设备连接成功后：
   a. 发现服务
   b. 绑定 Notify (0x150B)
   c. 写入 BF 默认值
   d. 加入 Coordinator 的设备列表（暂不启动 B0）
3. 用户确认通道映射，点击"开始播放"
4. Coordinator 统一启动 100ms 定时器
5. 所有设备开始接收 B0 指令
```
