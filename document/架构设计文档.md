## 1. 架构概述

### 1.1 架构风格

采用**分层架构 + 组件化模块**设计，自上而下分为 5 层：

```
┌──────────────────────────────────────────────────┐
│  UI 层         │  Kotlin / Jetpack Compose     │
│                │  黑色地牢主题 (Dark Dungeon)    │
├────────────────┼───────────────────────────────┤
│  应用层        │  Java                         │
│                │  ViewModel / UseCase / Manager │
├────────────────┼───────────────────────────────┤
│  领域层        │  Java                         │
│                │  Entity / Repository 接口     │
├────────────────┼───────────────────────────────┤
│  设备适配层    │  Java (新增)                   │
│                │  设备专属协议封装 / 周期调度    │
├────────────────┼───────────────────────────────┤
│  基础设施层    │  Java                         │
│                │  BLE/IO/Proto/音频引擎(AudioTrack)│
└────────────────┴───────────────────────────────┘
```

### 1.2 核心原则

| 原则 | 说明 |
|------|------|
| 单向依赖 | 上层依赖下层，下层不感知上层 |
| 接口隔离 | 每层通过接口通信，实现可替换 |
| 纯 Java 音频 | 音频播放不依赖 native 代码，使用 Android `AudioTrack` + `MediaCodec` 流式解码 |
| 响应式 UI | Compose + StateFlow 驱动界面更新 |
| 离线优先 | 所有配置/脚本/预设本地存储，不依赖网络 |

### 2.5 设备适配层 (Java, 新增)

**设计原则**：每种设备类型拥有独立的协议适配模块，**完全封装**其通信协议、数据格式、时序控制。上层只需知道"哪个通道在什么时间引用了哪段波形"，不关心波形内部是频率/强度还是其他格式。

#### 2.5.1 AdapterStatus — 适配器状态回调接口

```java
interface AdapterStatus {
    enum State { DISCONNECTED, CONNECTING, CONNECTED, RETRYING, ERROR }

    /** 连接状态变化时回调 */
    void onStateChanged(State state, String deviceId, String detail);

    /** 发送统计（每次输出周期后回调，用于调试和监控） */
    void onCycleStats(String deviceId, long writeLatencyMs, boolean success);

    /** 致命错误（如3次重连失败） */
    void onFatalError(String deviceId, String error);
}
```

#### 2.5.2 DeviceProtocolAdapter（最终版）

```java
interface DeviceProtocolAdapter {
    // ===== 标识 =====
    /** 设备类型标识（如 "coyote_v3", "coyote_v2", "lovense_vibrate"） */
    String getDeviceType();
    /** 设备实例唯一ID（用于日志/统计） */
    String getDeviceId();

    // ===== 生命周期 =====
    /** 建立连接（传输层细节由实现类封装，可以是BLE/USB/模拟） */
    void connect(android.content.Context context, String address, AdapterStatus status);
    /** 主动断开 */
    void disconnect();
    /** 销毁适配器，释放所有资源（定时器、BLE连接、缓冲区） */
    void release();

    // ===== 数据通道 =====
    /** 接收最新的时间轴快照数据（由 PlaybackCoordinator 推送，~33ms） */
    void updateSnapshot(Map<String, byte[]> channelData,
                        Map<String, Long> offsetsInSegment);
    /** seek 时清空内部缓冲区，立即用新位置数据覆盖 */
    void flush();
    /** 安全停止，强度归零 */
    void emergencyStop();

    // ===== 校验 =====
    /** 校验段数据是否属于本设备类型 */
    boolean validateSegmentData(byte[] protobufBytes);
}
```

**关键设计**：
- **传输层解耦**：`connect()` 接收 `Context + address` 字符串，不绑定 `BluetoothGatt`。BLE 实现类内部自行调用 `BluetoothAdapter.getRemoteDevice(address)`。未来 USB/网络设备只需不同的 `connect()` 实现
- **适配器自管理重连**：断连后适配器不销毁，自动尝试重连（最多 3 次，间隔递增），3 次失败后通过 `AdapterStatus.onFatalError()` 通知上层
- **状态上报**：每次周期输出后回调 `onCycleStats()` 报告写入延迟和成功率，便于调试和监控
- **V2 多特征值**：V2 适配器在 `connect()` 内部自行发现多个 GATT 特征值（PWM_AB2/A34/B34），接口层不关心

#### 2.5.3 设备子包封装原则

每种设备协议作为一个**完整自治的子包**封装在 `infrastructure/ble/adapter/{设备名}/` 下。架构层只关心子包对外的 Adapter 类（实现 `DeviceProtocolAdapter` 接口），**不干涉子包内部如何组织**。

**子包对外契约：**
- 子包根目录下必须存在至少一个 `public` 的 Adapter 类，实现 `DeviceProtocolAdapter`
- 子包内部的其他类（协议编解码、数据模型、常量、工具类）均为 `package-private`，不对外暴露
- 子包内部自主决定：类如何划分、是否使用内部状态机、是否拆分协议层等

**子包内部实现自治范围（不强制、不约束）：**
- 协议指令的编解码逻辑
- BLE GATT 连接/重连/定时器管理
- 内部数据模型（如波形帧、消息结构）
- 频率换算、强度映射等设备专属算法
- 流控状态机、缓冲区管理

> 各设备子包的具体实现细节，参见对应设备的协议移植方案文档。例如郊狼协议参见 [郊狼协议移植方案.md](郊狼协议移植方案.md)。

#### 2.5.4 DeviceTypeRegistry — 设备类型注册表

```java
class DeviceTypeRegistry {
    // 已注册的设备类型 → 适配器工厂
    private Map<String, Supplier<DeviceProtocolAdapter>> factories = new HashMap<>();

    /** 注册新设备类型 */
    void register(String deviceType, Supplier<DeviceProtocolAdapter> factory);

    /** 创建适配器实例 */
    DeviceProtocolAdapter create(String deviceType);

    /** 获取所有已注册类型 */
    Set<String> getRegisteredTypes();

    /** 获取设备类型的元信息（通道数、强度范围等） */
    DeviceTypeInfo getTypeInfo(String deviceType);
}

// 应用启动时注册
class AppModule {
    void init() {
        DeviceTypeRegistry registry = new DeviceTypeRegistry();
        registry.register("coyote_v3", CoyoteV3Adapter::new);
        registry.register("coyote_v2", CoyoteV2Adapter::new);
        registry.register("lovense_vibrate", LovenseAdapter::new);
        // 未来新增类型只需在此注册
    }

```

> 新增设备类型时，只需 (1) 编写 Adapter 实现、(2) 添加 `.proto`、(3) 在 `AppModule` 注册。UI 的设备类型选择列表以 `DeviceTypeRegistry.getRegisteredTypes()` 动态生成。

#### 2.5.5 PlaybackCoordinator — 设备协调器（拉取模型 + 线程安全）

```java
class PlaybackCoordinator {
    // 使用线程安全容器：Audio线程遍历 + BLE线程增删
    private final Map<String, CopyOnWriteArrayList<DeviceProtocolAdapter>> adaptersByType
        = new ConcurrentHashMap<>();

    /** 时间轴引擎每帧回调时调用（~33ms，Audio线程） */
    void onTimelineTick(long positionMs, TimelineSnapshot snapshot) {
        for (var entry : adaptersByType.entrySet()) {
            String deviceType = entry.getKey();
            CopyOnWriteArrayList<DeviceProtocolAdapter> adapters = entry.getValue();

            Map<String, byte[]> channelData = snapshot.getDataForDeviceType(deviceType);
            Map<String, Long> offsets = snapshot.getOffsetsForDeviceType(deviceType);

            for (DeviceProtocolAdapter adapter : adapters) {
                adapter.updateSnapshot(channelData, offsets);
            }
        }
    }

    /** 添加适配器（BLE线程） */
    void addAdapter(String deviceType, DeviceProtocolAdapter adapter) {
        adaptersByType.computeIfAbsent(deviceType,
            k -> new CopyOnWriteArrayList<>()).add(adapter);
    }

    /** 移除适配器（BLE线程，如3次重连失败后） */
    void removeAdapter(String deviceType, DeviceProtocolAdapter adapter) {
        CopyOnWriteArrayList<DeviceProtocolAdapter> list = adaptersByType.get(deviceType);
        if (list != null) {
            list.remove(adapter);
            adapter.release();
        }
    }

    /** seek + flushAll 原子操作 */
    void onSeek(long newPositionMs) { /* ... flush所有adapter */ }
    void emergencyStopAll() { /* ... */ }
}
```

> **设备容错**：单个设备断连不影响播放——音频和 TimelineEngine 继续运行，PlaybackCoordinator 继续向其他 Adapter 推送快照。仅断连的 Adapter 静默重连，重连成功后用 `flush()` 立即同步到当前播放位置。

## 3. 语言分工

### 3.1 语言分工

| 语言 | 占比 | 负责内容 |
|------|------|----------|
| **Java** | ~75% | 应用层、领域层、基础设施层(BLE/IO/音频引擎)、ViewModel |
| **Kotlin** | ~25% | UI 层(Compose)、主题、导航 |

### 3.2 音频引擎设计

音频引擎使用纯 Java 实现，不依赖 Native 代码或 JNI：

```
Java (PlaySessionVM)           Java (AudioEngine)
──────────────────────────     ────────────────────
play()  ──────────────────▶     AudioTrack.play()
pause() ──────────────────▶     AudioTrack.pause()
seek()  ──────────────────▶     extractor.seekTo() + flush()
getPosition() ────────────▶     seekOffset + headPosition
    
    ▲           │
    │  50ms 轮询 │  AudioTrack.getPlaybackHeadPosition()
    │           │  (采样级精度，无 native 回调)
    └───────────┘
```

**设计原则**：
- 流式解码：`MediaExtractor` 读取文件块 → `MediaCodec` 解码 → `AudioTrack.write()` 播放。内存占用 KB 级，不需要全量加载 PCM
- 进度控制：`AudioTrack.getPlaybackHeadPosition()` 返回已播放采样数，配合 seek offset 计算精确毫秒位置
- seek：`extractor.seekTo()` + `codec.flush()` + `audioTrack.flush()` + 重置 offset
- 多声道：`AudioTrack` 构造时传入源文件实际声道数，Android 音频框架自动 downmix 到输出设备

## 6. 关键设计决策

### 6.1 音频引擎为何用纯 Java (AudioTrack + MediaCodec)

| 决策 | 理由 |
|------|------|
| AudioTrack + MediaCodec | 流式解码播放，内存占用 KB 级，无需全量加载 PCM；进度精确到采样，多声道由系统自动 downmix |
| 自研播放器 | 需要 seek 时精确回溯时间轴，第三方播放器（ExoPlayer/MediaPlayer）的 seek 回调精度不足或行为不可控 |
| 纯 Java | 零 native 依赖，无 JNI/NDK/CMake/stl 兼容风险；编译部署简单，调试方便；延迟对本项目足够（不做实时乐器/低延迟监控） |

### 6.2 为何核心用 Java 而非纯 Kotlin

| 决策 | 理由 |
|------|------|
| BLE API | `android.bluetooth` 的 Java API 更稳定，Kotlin 协程封装存在兼容风险 |
| 团队熟悉度 | 核心业务逻辑团队更熟悉 Java |
| UI 用 Kotlin | Compose 是 Kotlin-first API，UI 层用 Kotlin 是自然选择 |

### 6.3 时间轴引擎职责简化

- 原设计：TimelineEngine (C) 直接计算强度值并输出 `Map<ch, Int>`
- 重构后：TimelineEngine 只做**段索引查询**——根据时间查找当前活跃的 `TimelineSegment`，输出 `TimelineSnapshot { segmentId, waveformData(bytes), offsetMs }`
- 强度/频率等具体计算**下移到设备适配层**，由各 Adapter 自行处理
- 代码简化：仅保留 `timeline_index`（段索引查找），移除 `timeline_calculator`

### 6.4 状态管理架构

```
UI (Compose)   ◄── StateFlow<UiState> ──  ViewModel (Java)
                                                │
                              ┌──────┬──────┬───┴───┐
                              │      │      │       │
                          Repository  Repository  Repository
```

- 每个 Screen 对应一个 ViewModel
- ViewModel 持有 `MutableStateFlow<UiState>`
- UI 通过 `collectAsState()` 订阅，自动重组
- 一次性事件（SnackBar/Navigate）用 `Channel<Event>` 投递

### 6.5 线程模型

| 线程 | 职责 | 优先级 |
|------|------|--------|
| **UI 线程** | Compose 渲染、用户交互 | 主线程 |
| **Audio 线程** | AudioTrack 写入 + MediaCodec 解码 | `THREAD_PRIORITY_AUDIO` |
| **BLE 线程** | GATT 写入、扫描回调 | `DEFAULT` |
| **Adapter 定时器** | 100ms/按需周期触发输出指令 | `DEFAULT` |
| **IO 线程池** (Java) | 文件读写、JSON/Proto 解析、预加载 | 后台 |
| **计算线程池** (Java) | 波形预览渲染、耗时校验 | 后台 |

### 6.6 为何用 Protobuf 而非 JSON 序列化波形数据

| 决策 | 理由 |
|------|------|
| 跨版本兼容 | Protobuf 自动忽略未知字段，新增字段不影响旧版本读取，JSON 需手动处理 |
| 紧凑高效 | 波形数据可能有数百个采样点，Protobuf 二进制比 JSON 小 3-5 倍 |
| 类型安全 | 生成强类型 Java 类，消除手动解析错误 |
| 外部 JSON 外壳 | 配置文件、播放列表仍用 JSON（数据量小、可手动编辑） |
| 不透明传输 | TimelineEngine 无需反序列化——只传 `bytes`，适配器自行解析 |

### 6.7 设备协议适配为何从单通道改为周期批量

| 决策 | 理由 |
|------|------|
| B0 双通道合并 | 郊狼 V3 的 B0 是 20 字节 A+B 合并指令，单通道 `encodeCommand()` 无法构造 |
| 100ms 固定窗口 | 协议要求每 100ms 发一条 B0，非按需逐通道发送 |
| 周期批量接口 | `sendCycle(Map<ch, bytes>)` 一次传入所有通道本周期数据 |
| 封装差异化 | 其他设备可能有不同的周期和指令结构，各自适配器自行处理 |

### 6.8 错误处理策略

```
┌───────────────────────┬──────────────────────────────┐
│ 错误类型               │ 处理方式                      │
├───────────────────────┼──────────────────────────────┤
│ 文件不存在             │ SnackBar + 标红曲目          │
│ 配置ID不匹配           │ 阻止添加 + Toast 提示         │
│ 时长不匹配             │ ⚠️ 图标标签，播放以音频时长为准 │
│ BLE 断连               │ 静默重连(不打断播放)，3次失败→│
│                       │ 提示用户+销毁适配器            │
│ 波形文件缺失           │ 静默跳过该段 + 日志            │
│ 音频解码失败           │ 跳过曲目 + 错误提示            │
│ Native 层错误          │ JNI 异常 → Java 捕获          │
└───────────────────────┴──────────────────────────────┘
```

### 6.9 后台播放保活

HypnoVibe 需要在屏幕关闭或切换到其他 APP 后继续播放音频并驱动设备。

#### 6.9.1 Android 前台服务 (Foreground Service)

```java
public class PlaybackService extends Service {
    private AudioEngine audioEngine;
    private PlaybackCoordinator coordinator;
    private NotificationManager notificationManager;

    @Override
    public int onStartCommand(Intent intent, int flags, int startId) {
        // 启动前台服务，必须附带 Notification
        startForeground(NOTIFICATION_ID, buildNotification());
        return START_STICKY;  // 被杀后自动重启
    }
}
```

#### 6.9.2 通知栏控制

```
┌──────────────────────────────────────────┐
│ HypnoVibe                                 │
│ 正在播放: 催眠引导                         │
│ ⏮  ▶  ⏭   郊狼 A:80 B:60                 │
└──────────────────────────────────────────┘
```

通知栏包含：
- 播放/暂停、上一首、下一首 按钮
- 当前曲目名称
- 郊狼通道实时强度
- 安全停止按钮（高强度 → 归零）

#### 6.9.3 架构适配

```
MainActivity ⇄⇄ PlaybackService (Foreground)
                     │
            ┌────────┼────────┐
            ▼        ▼        ▼
        AudioEngine Coordinator BLE Adapters
         (C native)  (Java)    (Java)
```

- 音频引擎和协调器**从 Activity 迁移到 Service** 中持有
- Activity 通过 `bindService()` 获取播放状态和控制器
- Service 负责 `startForeground()` 防止被系统杀死
- Android 13+ 需要 `POST_NOTIFICATIONS` 权限
- 唤醒锁 (WakeLock) 可选：仅在 B0 写入周期持有（~5ms/100ms），减小电量影响

#### 6.9.4 生命周期

```
用户点击播放:
  Activity.startService() → Service.startForeground()
  音频引擎启动 + Coordinator 启动 + Adapter 100ms 定时器启动

用户进入后台 (onStop):
  Activity unbindService()，Service 持续运行
  通知栏显示播放状态

用户回到前台 (onStart):
  Activity bindService()，恢复 UI 绑定

用户停止播放:
  Activity.stopService() → Service.stopForeground()
  音频引擎停止 → coordinator.emergencyStopAll() → 所有设备归零
```

> **注意**：安全停止必须是前台服务的最高优先级操作——即使 Service 被系统杀死，也要确保 `onDestroy()` 中发送归零指令。

## 7. Protobuf 序列化方案

### 7.1 设计原则

- 时间轴段中的波形数据使用 Protobuf 序列化，**每种设备类型定义独立的 message**
- Protobuf 的向后兼容性保证：新增字段不影响旧版本时间轴文件的读取
- TimelineEngine 不反序列化波形数据——字节流对其透明
- 只有设备适配器负责反序列化，适配器通过 `waveformType` 字段识别格式

### 7.2 Proto 文件定义

#### timeline_segment.proto

```protobuf
syntax = "proto3";
package hypnovibe;

message TimelineSegment {
  string segment_id = 1;
  string channel_id = 2;
  string label = 3;
  int64 start_time_ms = 4;
  int64 end_time_ms = 5;
  float strength_scale = 6;
  int32 loop_count = 7;          // -1=循环填充至endTimeMs
  bool disabled = 8;

  // 设备相关的波形数据（不透明字节流）
  bytes waveform_data = 9;

  // 标识波形数据的类型，适配器据此选择反序列化方式
  string waveform_type = 10;     // "coyote_v3_frame", "lovense_vibrate", etc.
}
```

#### coyote_waveform.proto

```protobuf
syntax = "proto3";
package hypnovibe;

// 郊狼 V3 波形数据：频率+强度 采样帧序列
message CoyoteV3Waveform {
  repeated WaveFrame frames = 1;
  int32 sample_rate = 2;         // 帧/秒（如 20 表示每 50ms 一帧）
  string source_format = 3;      // "dg_lab_v3", "custom"
  int32 total_duration_ms = 4;
}

message WaveFrame {
  int32 time_ms = 1;             // 帧内相对时间(ms)
  int32 frequency = 2;           // 波形频率 10-1000 (用户值，适配器内换算)
  int32 strength = 3;            // 波形强度 0-100
}

// 郊狼 V2 波形数据：X/Y/Z 参数序列
message CoyoteV2Waveform {
  repeated V2WaveFrame frames = 1;
  int32 total_duration_ms = 2;
}

message V2WaveFrame {
  int32 time_ms = 1;
  int32 x_value = 2;             // 连续脉冲数 0-31
  int32 y_value = 3;             // 脉冲间隔 0-1023
  int32 z_value = 4;             // 脉冲宽度 0-31
}
```

#### lovense_waveform.proto

```protobuf
syntax = "proto3";
package hypnovibe;

message LovenseVibrateWaveform {
  repeated VibratePoint points = 1;
}

message VibratePoint {
  int32 time_ms = 1;
  int32 strength = 2;            // 0-20
}
```

### 7.3 版本兼容策略

```
场景：用户用 v1.0 APP 创建了时间轴
      v1.0 APP 更新到 v2.0（新增了字段）

  v1.0 时间轴文件:
    CoyoteV3Waveform { frames: [...], sample_rate: 20 }

  v2.0 新增字段:
    CoyoteV3Waveform { frames: [...], sample_rate: 20, channel_strength: 150 }
                                                          ^^^^^^^^^^^^^^^^ 新增

  v1.0 APP 读取 v2.0 文件:
    Protobuf 自动忽略未知字段 channel_strength，读取 frames + sample_rate 正常

  v2.0 APP 读取 v1.0 文件:
    channel_strength 字段缺失 → 使用默认值 (0)，不报错
```

### 7.4 与现有 JSON 文件的关系

- 设备配置 (`.hvconfig`)：保持 JSON，结构简单无需 Protobuf
- 播放列表 (`.hvplaylist`)：保持 JSON
- 时间轴脚本 (`.hvscript`)：JSON 外壳，`segments[].waveformData` 字段存储 **Base64 编码的 Protobuf 字节**
- 波形文件 (`.hvwav`)：JSON 外壳，`dataPoints` 字段存储 Base64 编码的 Protobuf 字节

> 不直接用 `.pb` 二进制文件是为了保持文件可手动查看/编辑（JSON 可读），同时波形数据部分用 Protobuf 保证扩展性。

### 7.5 protoc 版本锁定

为避免不同开发者/CI 环境的 protoc 版本不一致导致生成的 Java 类不兼容，项目内锁定 protoc 版本：

```
tools/
└── protoc/
    ├── protoc-28.0-linux-x86_64    # Linux CI
    ├── protoc-28.0-windows-x86_64  # Windows 开发
    └── protoc-28.0-osx-x86_64      # macOS 开发

app/build.gradle.kts:
  protobuf {
      protoc {
          artifact = "com.google.protobuf:protoc:3.21.12"  // 官方 Maven
      }
  }
```

> 首选通过 Gradle protobuf 插件自动下载，备选在 `tools/protoc/` 目录存放二进制文件供离线环境使用。

## 附录 A：AndroidManifest 关键声明

### 权限

```xml
<!-- 蓝牙 (Android 12+) -->
<uses-permission android:name="android.permission.BLUETOOTH_SCAN" />
<uses-permission android:name="android.permission.BLUETOOTH_CONNECT" />
<!-- 蓝牙 (Android 11 及以下) -->
<uses-permission android:name="android.permission.BLUETOOTH"
    android:maxSdkVersion="30" />
<uses-permission android:name="android.permission.BLUETOOTH_ADMIN"
    android:maxSdkVersion="30" />
<!-- 定位 (BLE 扫描需要，Android 11-) -->
<uses-permission android:name="android.permission.ACCESS_FINE_LOCATION"
    android:maxSdkVersion="30" />
<!-- 前台服务 -->
<uses-permission android:name="android.permission.FOREGROUND_SERVICE" />
<uses-permission android:name="android.permission.FOREGROUND_SERVICE_MEDIA_PLAYBACK" />
<!-- 通知 (Android 13+) -->
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
<!-- 唤醒锁 (可选，后台保持 CPU 唤醒) -->
<uses-permission android:name="android.permission.WAKE_LOCK" />
<!-- 读取音频文件 -->
<uses-permission android:name="android.permission.READ_EXTERNAL_STORAGE"
    android:maxSdkVersion="32" />
<uses-permission android:name="android.permission.READ_MEDIA_AUDIO" />
<!-- 管理所有文件 (Android 11+) -->
<uses-permission android:name="android.permission.MANAGE_EXTERNAL_STORAGE"
    tools:ignore="ScopedStorage" />
```

### Service 声明

```xml
<application>
    <service
        android:name=".service.PlaybackService"
        android:exported="false"
        android:foregroundServiceType="mediaPlayback" />
</application>
```

### 运行时权限

| 权限 | 时机 | 必要性 |
|------|------|:--:|
| `MANAGE_EXTERNAL_STORAGE` | Android 11+，APP 启动时跳系统设置 | 必须 |
| `BLUETOOTH_SCAN/CONNECT` | 首次进入设备管理页面 | 必须 |
| `ACCESS_FINE_LOCATION` | Android ≤11，首次扫描前 | 必须 |
| `POST_NOTIFICATIONS` | Android 13+，首次播放前 | 必须 |
| `WAKE_LOCK` | 无运行时弹窗，直接声明即可 | 可选 |
